A 4e increment adds =defmacro= support by *selective pre-expansion via
SBCL* (=ldb-cl-macro.el=; strategy locked in =01-overview.org= L3).  The
bridge does not translate macros directly; instead an SBCL code-walker
expands only /user/ macros (operators whose =macro-function= is non-nil
and whose home package is not =COMMON-LISP=), one =macroexpand-1= step at
a time, leaving standard CL macros (=when=/=loop=/=defun=/=case=/...) for
the existing 4a-4d' handlers.  =defmacro= forms are evaluated into SBCL
then dropped; output is printed =*print-case* :downcase= (no
length/level truncation) so the upcased CL symbols round-trip to the
lowercase symbols the reader dispatches on.  Public entries:
=ldb-cl-macro-expand-string= (CL → expanded CL text),
=ldb-cl-translate-string-expanding= / =-file-expanding= (pre-expand then
translate).  Gated on SBCL on =PATH=; without it =defmacro= stays
rejected loudly.  Validated differentially (=test/ldb-cl-macro-test.el=,
folded into =make difftest=): five macro programs — a simple
=`(* ,x ,x)=, a =&body= macro, a macro that /generates a defun/, nested
user macros, and a macro used inside =loop= — all agree with SBCL
running the original; =make difftest= 20/20.  A surviving backquote in a
non-macro form is harmless: SBCL's reader lowers =`(a ,b)= to standard
=(list 'a b)= before it reaches the bridge.  Out for v1:
=macrolet=/=symbol-macrolet= (local macros — the walker uses only the
global =macro-function=), compiler-macros, read-macros beyond
=#\\=/=#'=/=#|...|#=.  Hygiene is faithful-to-oracle: we reproduce
exactly SBCL's (unhygienic) expansion, so capture behaviour matches the
CL program and the differential test is the gate.

* Why CL-core first (vs roadmap's Scheme Phase 2)

The end goal that motivated this work is a self-improving agent that
stands on decades of Lisp AI heritage (genetic programming, symbolic
AI).  That heritage is overwhelmingly *Common Lisp*.  CL is also a
*Lisp-2* like Elisp, so cl-lib / eieio already carry most of the
runtime — translation is near-identity for the core.  Scheme's
Lisp-1 + hygienic macros + call/cc make it the harder, less
AI-relevant target.  So we front-load CL core; the dialect-neutral IR
means Scheme still benefits later.

Lambda-list handling: a CL λ-list needs =cl-defun=/=cl-function= iff it
contains =&key=, =&aux=, or an =&optional=/=&rest= var with a default
or supplied-p.  Plain =(a b &optional c &rest d)= → plain =defun=.

Declarations =(declare …)= and =(the TYPE expr)= are *stripped*
(=the= keeps its expr) — they are hints, semantically droppable.

* Scope — OUT (reject loudly, =ldb-cl-unsupported-form-error=)

condition definitions
(=define-condition/handler-bind/restart-case=), macros
(=defmacro/macrolet/define-symbol-macro=), packages
(=in-package/defpackage=), I/O with divergent semantics
(=print/princ/write/read=), =tagbody/go=, =eval-when=,
nested backquote, =call/cc=-equivalents, binding/clause forms
(=destructuring-bind/with-slots/with-accessors=).  Each signals with
the offending form — never silent.  These land in later increments
(macros gated on the L3 hygiene decision).

Rationale: a narrow, *correct* core that runs real algorithms beats a
broad, subtly-wrong translator.  Matches CLAUDE.md invariant 4.

expression, pass the (possibly remapped) head through.  =ref= nodes
emit a bare symbol, so =setq=/=block= names round-trip correctly.

#+begin_src text
  CL source string

* Known hazards

- HZ1. *Class / accessor names that shadow Emacs core functions.*  A CL
  =(defclass point ...)= translates to an eieio =defclass=.  With
  =eieio-backward-compatibility= (the Emacs default) eieio defines a
  constructor *function* named after the class, so =point= is fset to a
  constructor and the built-in =(point)= is clobbered — later host code
  that calls =(point)= for a buffer position then receives an eieio
  object and errors far from the translation site.  Discovered by the
  SBCL differential suite (the CLOS case crashed ERT's own message-log
  truncation).  Same risk for any class / accessor / =defstruct=
  constructor whose name collides with a core function (=point=, =car=
  via accessor, etc.).  Mitigations to weigh in a later increment:
  (a) emit =defclass= with =eieio-backward-compatibility= locally
  disabled or with no implicit constructor; (b) warn (=:warnings=) when
  a translated definition name shadows an =fboundp= core symbol;
  (c) leave it as a documented caller responsibility.  For now: loud
  documentation + test classes avoid core names.

* Open questions

- OQ1. =defstruct= → =cl-defstruct= — DONE (4b).  Name/option spec is
  kept verbatim (cl-defstruct accepts the same =(:constructor ...)= etc.),
  slot defaults are translated.  Exotic =:type vector= layouts untested.
- OQ2. =format= directive translation — DONE (4b) for the basic subset
  (=~a ~s ~d ~f ~e ~g ~x ~o ~c ~% ~& ~~=, destinations =nil= / =t=).
  Parameterized (=~5d=, =~,2f=), iteration (=~{...~}=) and conditional
  (=~[...~]=) directives are rejected loudly; a later increment can add
  a real directive parser.
- OQ3. Macro translation — RESOLVED (4e).  Selective pre-expansion via
  SBCL (=ldb-cl-macro.el=): expand only user macros, drop =defmacro=,
  keep standard macros for the existing handlers.  Native hygienic
  =syntax-rules= in Elisp (=01-overview.org= L3 "Phase 4+") remains a
  separate, deferred option for an SBCL-less path.
