* Why Scheme is the harder target (vs CL near-identity)

CL → Elisp is near-identity (both Lisp-2).  Scheme differs in ways that
need real translation, not symbol remapping:

1. *Lisp-1 vs Lisp-2* — Scheme has ONE namespace; Elisp has two.  A bound
   variable can be called like a function (=(f x)= where =f= is a
   parameter).  This is the central problem (see L-SCM1).
2. *Booleans / truthiness* — Scheme: =#f= is the ONLY false value; =()=,
   =0=, =""= are all TRUE.  Elisp: =nil= (= =()=) is false.  See L-SCM2.
3. *Tail calls* — Scheme guarantees TCO; Elisp does not.  Deep
   tail-recursion that a Scheme relies on may overflow in Elisp.  See
   L-SCM3.
4. *Naming* — predicates end =?= (=null?=, =eq?=), mutators end =!=
   (=set!=, =set-car!=), booleans =#t=/=#f=.  Pure remap (easy part).

* Locked decisions

** L-SCM1. Lisp-1 → Lisp-2 = scope-tracked funcall insertion

The translator threads a lexical environment =ENV= (the set of
locally-bound symbols: =lambda= params, =let=/=let*=/=letrec= vars,
named-let loop var + params, internal =define=).  A pre-pass over the
top-level forms classifies each global =define= as a /function/
(=(define (f …) …)= or =(define f (lambda …))=) or a /variable/
(=(define x VAL)=, VAL not a lambda).

- *Call position* =(OP . ARGS)=, OP a symbol, not a special form:
  - OP ∈ ENV (a local binding) → =(funcall OP . ARGS')= — it holds a
    procedure value.
  - else → =(OP' . ARGS')= where OP' is the remapped global/builtin.
- *Value position* (symbol NOT in call position):
  - sym ∈ ENV → =sym= (a value).
  - sym is a known function (global function from the pre-pass, or a
    builtin in the remap/known-function table) → =#'sym'= (the function
    cell as a value — needed for =(map f xs)= etc.).
  - sym is a global /variable/ from the pre-pass → =sym=.
  - otherwise → =sym= (unknown; differential test will catch a wrong
    guess; we do not silently =#'= an unknown).

=funcall= is represented with the existing =call= IR tag (=:fn 'funcall=,
=:args (OP . ARGS')=); =#'sym= reuses the =function=/quote machinery —
no new IR top-level keys (CLAUDE.md invariant 2).

** L-SCM2. Booleans = =#t=→t, =#f=→nil; ()-as-true is a documented gap

The reader maps =#t=→=t=, =#f=→=nil=.  Real Scheme overwhelmingly uses
=#f= for false and does not rely on =()= / =0= being true in test
position, so ordinary conditionals translate correctly.  The residual
gap — =(if '() a b)= is =a= in Scheme but =b= in Elisp — is
context-dependent and statically undetectable, so it is a *documented
fidelity gap* (same class as CL's single-value =values= leak), not a
loud reject.  Predicate results (=null?= etc. → t/nil) are exact.

** L-SCM3. Tail calls = best-effort; deep TCO loops documented

=named-let= → Emacs =named-let= (28+), =letrec= → =letrec=.  Elisp does
not guarantee TCO, so a Scheme that loops via deep tail recursion can
overflow =max-lisp-eval-depth=.  v1 does not transform tail recursion
into iteration; this is a documented limitation.  (A later increment
could special-case tail-self-calls in =named-let= into =while=.)

** L-SCM4. Subset, reject loudly (mirrors CL invariant 4)

Continuations (=call/cc=, =call-with-current-continuation=,
=dynamic-wind=), hygienic macros (=define-syntax=/=syntax-rules= — the
Scheme analogue of CL macros; a later increment can pre-expand via Guile,
mirroring the SBCL path), multiple values (=values=/=call-with-values= —
same cl-lib limit as CL), full numeric tower (rationals/complex), vectors
=#(…)=, and ports/IO beyond =display=/=newline= are OUT and signal
=ldb-scheme-unsupported-form-error=.
