* What is lisp-dialect-bridge?

A *NeLisp extension package* that translates between sibling Lisp
dialects and Emacs Lisp through a shared IR.  Phase 1 ships a
unidirectional Guix Scheme → Elisp recipe importer; later phases
generalise to R7RS Scheme, Common Lisp subset, Nix expression, and
back.

#+begin_quote
The same trick SBCL pulled (CL implemented in CL): translate the
parts of sibling Lisps that matter into the substrate where Emacs
itself lives.  Then anvil-pkg can ingest Guix recipes, NeLisp can
host Common Lisp libraries, and AI agents speak one Lisp through the
others.
#+end_quote

* Status

*Phase 1 SHIPPED* (2026-04-27).  Guix recipe importer is live; ERT
9/9 PASS (mocked, no Scheme implementation needed).  Public entry:

#+begin_src elisp
(ldb-guix-import-string SOURCE NAME-SYM)  ; from string
(ldb-guix-import-file PATH NAME-SYM)      ; from file
(ldb-guix-import-form FORM)               ; from already-read S-exp
#+end_src

Output is a =pkg-define= S-expression (data — caller pp / write /
eval).  Phase 2 (general R7RS subset translator) is the next
milestone; see =docs/design/01-overview.org=.

* Why this exists

Three independent reasons compound:

1. *Guix migration moat.* anvil-pkg's biggest user-acquisition risk
   is "Guix has 30 000 Scheme recipes I do not want to rewrite".  A
   working Scheme-recipe → anvil-pkg-define translator removes that
   risk.
2. *Ecosystem import.*  Common Lisp has decades of mature libraries
   (asdf, alexandria, clack, hunchentoot, ...).  Translating a
   useful subset to Elisp gives NeLisp a credible standard library
   without re-authoring everything.
3. *Polyglot Lisp Machine path.*  The strategy-vision doc in
   anvil-pkg argues that Elisp + NeLisp + bridge approaches a "world
   class Lisp" by importing what the other dialects do better
   (Racket's hygienic macros, CL's CLOS, Clojure's persistent
   collections).  This repo is where that import happens.

* Goals

- Translate constrained subsets of /Scheme/, /Common Lisp/, and the
  /Nix expression language/ into Elisp source (uni-directional MVP).
- Extend to bidirectional Elisp ↔ Scheme once the IR is stable.
- Stay a clean NeLisp extension — no fork of NeLisp internals, no
  monkey-patching.
- Keep the IR small enough that one human can reason about every
  node type at once.

* Non-goals

- Translating /every/ Scheme / CL / Clojure program.  We translate
  /useful subsets/ and reject pathological forms loudly.
- A Common Lisp implementation in Elisp.  cl-lib already exists; we
  are about /translation/, not implementation.
- A Nix evaluator in Elisp.  We translate Nix source, then hand to
  anvil-pkg-render-nix or the Nix daemon.
- Continuations / =call/cc= for arbitrary control flow.  Bridge will
  reject programs that depend on first-class continuations (a small
  minority of real-world Scheme).

* Approach overview

The CL-core translator (4a–4d') is validated against *SBCL as a
ground-truth oracle*: =make difftest= (=test/ldb-cl-sbcl-diff.el=) runs
each program both in SBCL and through the bridge and asserts the printed
results agree.  =make test= stays hermetic; the differential suite skips
cleanly when =sbcl= is not on =PATH=.

Phase 4e adds Common Lisp =defmacro= support by *selective
pre-expansion* through SBCL (=ldb-cl-macro.el=) rather than translating
macros directly: an SBCL walker expands only user macros (leaving
standard =when=/=loop=/=defun=/... for the normal handlers), drops the
=defmacro= forms, and emits downcased CL the existing translator
ingests.  Use =ldb-cl-translate-string-expanding= /
=ldb-cl-translate-file-expanding= for sources that define and use
macros.  Correctness is gated by the same SBCL differential discipline.

This pipeline is validated on a *real CL library* — verbatim definitions
from =alexandria= (=test/ldb-cl-alexandria-test.el=): the functions
=ensure-list=, =flatten=, =iota=, =clamp=, =proper-list-p= (a =do= loop),
the binding macros =if-let= / =when-let= / =when-let*=, and =xor= (a named
=block= built by alexandria's own =with-gensyms=, returning multiple
=values=, read through =multiple-value-list=) translate and run
identically to SBCL.  Documented frontier: =multiple-value-call= (broken
in Emacs cl-lib, rejected loudly) and the single-value-context =values=
leak.  =make difftest= 48/48.

Phase 2 adds a *Scheme core-syntax* translator (=ldb-scheme-cvt.el=;
=ldb-scheme-translate-string= / =-file=).  Unlike CL (Lisp-2, near
identity), Scheme is Lisp-1 — one namespace — so the translator tracks
lexical scope and a top-level define pre-pass to insert =funcall= where a
bound variable is called like a function, and =#'fn= where a global
function is passed by value.  =define= / =lambda= / =let= / =let*= /
=letrec= / named-let / =if= / =cond= / =set!= / quasiquote
=`(a ,b ,@c)= / predicates (=?=) and mutators (=!=) map to their Elisp
equivalents; =#t='t, =#f='nil.  Hygienic macros (=define-syntax= /
=syntax-rules=) are supported via *Guile pre-expansion*
(=ldb-scheme-translate-string-expanding=, =ldb-scheme-macro.el= — the
SBCL-path analogue): Guile expands the macros to readable Scheme via
=tree-il->scheme= (hygiene renames and all), which then translates.
Continuations (=call/cc=) and multiple values are rejected loudly.  Correctness
is gated by *GNU Guile 3.0 as the differential oracle*
(=test/ldb-scheme-guile-diff.el=, =make schemediff=): higher-order
=funcall=, applied closures, =letrec= mutual recursion, named-let loops
and more all run identically to Guile.  It is further validated on *real
Scheme* — nine verbatim *SICP* procedures (=test/ldb-scheme-sicp-test.el=):
Newton's-method =sqrt=, higher-order =sum= / =accumulate=, tree-recursive
=count-change=, and the symbolic-differentiation program =deriv= (SICP
2.3.2) all match Guile, plus =define-syntax= macros (=swap!= with
hygiene, =while=, ellipsis) via Guile pre-expansion.  =make schemediff=
39/39.

* MCP integration (optional)

=ldb-mcp.el= is an *optional* adapter that exposes the translators as
anvil MCP tools on the =emacs-eval= server, so an agent (Claude Code) can
call them directly instead of shelling out to =emacs --batch=.  It is the
only file that requires anvil; the core translators stay anvil-free
(invariant 5), and it is *not* in the =make compile= SRC list (it loads
only inside a running anvil daemon).

Tools:

| Tool                    | Purpose                                                |
|-------------------------+--------------------------------------------------------|
| =ldb-translate=         | dialect + source/path [+ expand + pretty] → Elisp text |
| =ldb-translate-to-file= | dialect + out + source/path [+ expand] → write a =.el= |
| =ldb-capabilities=      | (no args) dialects / oracle presence / live rejects    |

=expand="1"= pre-expands macros first (=defmacro= via SBCL for CL,
=define-syntax= via Guile for Scheme).  Enable inside the daemon:

#+begin_src elisp
(add-to-list 'load-path "/path/to/lisp-dialect-bridge")
(require 'lisp-dialect-bridge)
(load "/path/to/lisp-dialect-bridge/ldb-mcp.el")
(ldb-mcp-enable)
#+end_src

A client picks the tools up on its next =tools/list= fetch (a fresh
session); add the snippet to the daemon init to persist across restarts.

* Naming

=lisp-dialect-bridge= is the working name.  Per the anvil ecosystem
strategy doc (=anvil-pkg/docs/design/03-strategy-vision.org=) the
project may later rename to =anvil-bridge= for cohesion.  We revisit
that decision when Phase 1 ships and the project's identity is clear.
